# Claude Code 세션 팀 조율

> 공유 태스크, 에이전트 간 메시지, 중앙화된 관리를 통해 여러 Claude Code 인스턴스를 팀으로 조율합니다.

> **경고**: Agent teams는 실험적 기능이며 기본적으로 비활성화되어 있습니다. [settings.json](/en/settings) 또는 환경 변수에 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS`를 추가하여 활성화할 수 있습니다. Agent teams에는 세션 재개, 태스크 조율, 종료 동작과 관련된 [알려진 제한 사항](#limitations)이 있습니다.

Agent teams를 사용하면 여러 Claude Code 인스턴스를 함께 조율할 수 있습니다. 하나의 세션이 팀 리드 역할을 맡아 작업을 조율하고, 태스크를 할당하며, 결과를 종합합니다. 팀원들은 각자의 컨텍스트 창에서 독립적으로 작업하며 서로 직접 소통합니다.

[서브에이전트](/en/sub-agents)와 달리, 서브에이전트는 단일 세션 내에서 실행되며 메인 에이전트에게만 결과를 보고할 수 있는 반면, agent teams에서는 리드를 거치지 않고 개별 팀원과 직접 상호작용할 수 있습니다.

* [팀 시작하기](#start-your-first-agent-team)
* [팀원 제어](#control-your-agent-team) - 표시 모드, 태스크 할당, 위임 포함
* [병렬 작업 모범 사례](#best-practices)

## Agent teams를 사용할 시기

Agent teams는 병렬 탐색이 실질적인 가치를 더하는 태스크에 가장 효과적입니다. 전체 시나리오는 [활용 사례 예시](#use-case-examples)를 참조하세요. 가장 강력한 활용 사례는 다음과 같습니다:

* **연구 및 검토**: 여러 팀원이 문제의 서로 다른 측면을 동시에 조사하고, 발견 사항을 공유하고 서로 검증할 수 있습니다
* **새 모듈 또는 기능**: 팀원들이 서로 방해하지 않고 각자 별도의 작업을 담당할 수 있습니다
* **경쟁 가설을 통한 디버깅**: 팀원들이 서로 다른 이론을 병렬로 테스트하여 더 빠르게 답을 찾을 수 있습니다
* **크로스 레이어 조율**: 프론트엔드, 백엔드, 테스트에 걸친 변경 사항을 각각 다른 팀원이 담당합니다

Agent teams는 조율 오버헤드가 추가되며 단일 세션보다 훨씬 많은 토큰을 사용합니다. 팀원들이 독립적으로 작동할 수 있을 때 가장 잘 작동합니다. 순차적 태스크, 동일 파일 편집, 또는 의존성이 많은 작업에는 단일 세션이나 [서브에이전트](/en/sub-agents)가 더 효과적입니다.

### 서브에이전트와의 비교

Agent teams와 [서브에이전트](/en/sub-agents) 모두 작업을 병렬화할 수 있지만 다르게 작동합니다. 작업자들이 서로 소통해야 하는지 여부에 따라 선택하세요:

|                   | 서브에이전트                                        | Agent teams                                         |
| :---------------- | :----------------------------------------------- | :-------------------------------------------------- |
| **컨텍스트**       | 자체 컨텍스트 창; 결과가 호출자에게 반환됨 | 자체 컨텍스트 창; 완전히 독립적               |
| **소통**          | 메인 에이전트에게만 결과 보고       | 팀원들이 서로 직접 메시지 전송               |
| **조율**          | 메인 에이전트가 모든 작업 관리                      | 공유 태스크 목록으로 자체 조율             |
| **최적 용도**      | 결과만 중요한 집중적인 태스크      | 논의와 협업이 필요한 복잡한 작업 |
| **토큰 비용**    | 낮음: 결과가 메인 컨텍스트로 요약됨   | 높음: 각 팀원이 별도의 Claude 인스턴스 |

빠르고 집중적인 작업자가 필요할 때는 서브에이전트를 사용하세요. 팀원들이 발견 사항을 공유하고, 서로 검증하고, 자체적으로 조율해야 할 때는 agent teams를 사용하세요.

## Agent teams 활성화

Agent teams는 기본적으로 비활성화되어 있습니다. 셸 환경 또는 [settings.json](/en/settings)에서 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` 환경 변수를 `1`로 설정하여 활성화합니다:

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

## 첫 번째 agent team 시작하기

Agent teams를 활성화한 후, Claude에게 agent team을 만들고 원하는 태스크와 팀 구조를 자연어로 설명하면 됩니다. Claude가 팀을 만들고, 팀원을 생성하며, 프롬프트에 따라 작업을 조율합니다.

다음 예시는 세 역할이 서로를 기다리지 않고 독립적으로 문제를 탐색할 수 있어 잘 작동합니다:

```
코드베이스 전체의 TODO 주석을 추적하는 CLI 도구를 설계 중입니다.
다양한 관점에서 탐색하는 agent team을 만들어 주세요: 한 팀원은
UX를, 한 팀원은 기술 아키텍처를, 한 팀원은 악마의 대변인 역할을 맡아주세요.
```

그러면 Claude가 [공유 태스크 목록](/en/interactive-mode#task-list)으로 팀을 만들고, 각 관점을 위한 팀원을 생성하고, 문제를 탐색하게 하고, 발견 사항을 종합한 후, 작업이 완료되면 [팀을 정리](#clean-up-the-team)하려고 시도합니다.

리드의 터미널에는 모든 팀원과 그들이 작업 중인 내용이 나열됩니다. Shift+Up/Down을 사용하여 팀원을 선택하고 직접 메시지를 보낼 수 있습니다.

각 팀원을 별도의 분할 창에 표시하려면 [표시 모드 선택](#choose-a-display-mode)을 참조하세요.

## Agent team 제어

리드에게 원하는 내용을 자연어로 알려주세요. 리드는 지시에 따라 팀 조율, 태스크 할당, 위임을 처리합니다.

### 표시 모드 선택

Agent teams는 두 가지 표시 모드를 지원합니다:

* **인프로세스**: 모든 팀원이 메인 터미널 내에서 실행됩니다. Shift+Up/Down으로 팀원을 선택하고 타이핑하여 직접 메시지를 보낼 수 있습니다. 별도의 설정 없이 모든 터미널에서 작동합니다.
* **분할 창**: 각 팀원이 자체 창을 갖습니다. 모든 사람의 출력을 동시에 볼 수 있고 창을 클릭하여 직접 상호작용할 수 있습니다. tmux 또는 iTerm2가 필요합니다.

> **참고**: `tmux`는 특정 운영 체제에서 알려진 제한 사항이 있으며 전통적으로 macOS에서 가장 잘 작동합니다. iTerm2에서 `tmux -CC`를 사용하는 것이 `tmux`의 권장 진입점입니다.

기본값은 `"auto"`로, 이미 tmux 세션 내에서 실행 중이면 분할 창을 사용하고 그렇지 않으면 인프로세스를 사용합니다. `"tmux"` 설정은 분할 창 모드를 활성화하며 터미널을 기반으로 tmux 또는 iTerm2 사용 여부를 자동 감지합니다. 재정의하려면 [settings.json](/en/settings)에서 `teammateMode`를 설정하세요:

```json
{
  "teammateMode": "in-process"
}
```

단일 세션에서 인프로세스 모드를 강제하려면 플래그로 전달하세요:

```bash
claude --teammate-mode in-process
```

분할 창 모드에는 [tmux](https://github.com/tmux/tmux/wiki) 또는 [`it2` CLI](https://github.com/mkusaka/it2)가 있는 iTerm2가 필요합니다. 수동으로 설치하려면:

* **tmux**: 시스템 패키지 관리자를 통해 설치하세요. 플랫폼별 지침은 [tmux 위키](https://github.com/tmux/tmux/wiki/Installing)를 참조하세요.
* **iTerm2**: [`it2` CLI](https://github.com/mkusaka/it2)를 설치한 후 **iTerm2 → 설정 → 일반 → 매직 → Python API 활성화**에서 Python API를 활성화하세요.

### 팀원 및 모델 지정

Claude가 태스크에 따라 생성할 팀원 수를 결정하거나, 원하는 내용을 정확히 지정할 수 있습니다:

```
4명의 팀원으로 팀을 만들어 이 모듈들을 병렬로 리팩터링해 주세요.
각 팀원에게 Sonnet을 사용해 주세요.
```

### 팀원의 계획 승인 요구

복잡하거나 위험한 태스크의 경우, 팀원이 구현 전에 계획을 세우도록 요구할 수 있습니다. 팀원은 리드가 접근 방식을 승인할 때까지 읽기 전용 계획 모드에서 작업합니다:

```
인증 모듈을 리팩터링할 아키텍트 팀원을 생성해 주세요.
변경하기 전에 계획 승인을 요구해 주세요.
```

팀원이 계획을 완료하면 리드에게 계획 승인 요청을 보냅니다. 리드는 계획을 검토하고 승인하거나 피드백과 함께 거부합니다. 거부되면 팀원은 계획 모드에 남아 피드백을 기반으로 수정하고 재제출합니다. 승인되면 팀원은 계획 모드를 종료하고 구현을 시작합니다.

리드는 자율적으로 승인 결정을 내립니다. 리드의 판단에 영향을 미치려면 "테스트 커버리지를 포함하는 계획만 승인" 또는 "데이터베이스 스키마를 수정하는 계획은 거부"와 같은 기준을 프롬프트에 제공하세요.

### 위임 모드 사용

위임 모드 없이는 리드가 팀원을 기다리는 대신 직접 태스크를 구현하기 시작하는 경우가 있습니다. 위임 모드는 리드를 조율 전용 도구(팀원 생성, 메시지 전송, 종료, 태스크 관리)로만 제한하여 이를 방지합니다.

이는 리드가 코드를 직접 건드리지 않고 작업 분해, 태스크 할당, 결과 종합과 같은 오케스트레이션에만 집중하도록 하려는 경우에 유용합니다.

활성화하려면 먼저 팀을 시작한 다음 Shift+Tab을 눌러 위임 모드로 전환하세요.

### 팀원과 직접 대화

각 팀원은 완전하고 독립적인 Claude Code 세션입니다. 팀원에게 직접 메시지를 보내 추가 지시를 내리거나, 후속 질문을 하거나, 접근 방식을 바꿀 수 있습니다.

* **인프로세스 모드**: Shift+Up/Down으로 팀원을 선택하고 타이핑하여 메시지를 보내세요. Enter를 눌러 팀원의 세션을 보고, Escape로 현재 턴을 중단하세요. Ctrl+T를 눌러 태스크 목록을 전환하세요.
* **분할 창 모드**: 팀원의 창을 클릭하여 세션과 직접 상호작용하세요. 각 팀원은 자체 터미널의 전체 보기를 갖습니다.

### 태스크 할당 및 클레임

공유 태스크 목록이 팀 전체의 작업을 조율합니다. 리드가 태스크를 만들고 팀원들이 처리합니다. 태스크는 보류 중, 진행 중, 완료의 세 가지 상태를 가집니다. 태스크는 다른 태스크에 의존할 수도 있습니다. 미해결 의존성이 있는 보류 중 태스크는 해당 의존성이 완료될 때까지 클레임할 수 없습니다.

리드는 태스크를 명시적으로 할당하거나 팀원들이 자체 클레임할 수 있습니다:

* **리드 할당**: 리드에게 어떤 태스크를 어떤 팀원에게 줄지 알려주세요
* **자체 클레임**: 태스크를 완료한 후 팀원이 다음 미할당, 비차단 태스크를 자체적으로 가져갑니다

태스크 클레임은 여러 팀원이 동시에 같은 태스크를 클레임하려 할 때 경쟁 조건을 방지하기 위해 파일 잠금을 사용합니다.

### 팀원 종료

팀원의 세션을 정상적으로 종료하려면:

```
researcher 팀원에게 종료를 요청해 주세요
```

리드가 종료 요청을 보냅니다. 팀원은 승인하여 정상 종료하거나 설명과 함께 거부할 수 있습니다.

### 팀 정리

작업이 완료되면 리드에게 정리를 요청하세요:

```
팀을 정리해 주세요
```

이렇게 하면 공유 팀 리소스가 제거됩니다. 리드가 정리를 실행하면 활성 팀원이 있는지 확인하고 아직 실행 중인 팀원이 있으면 실패하므로, 먼저 팀원들을 종료하세요.

> **경고**: 항상 리드를 통해 정리하세요. 팀원의 팀 컨텍스트가 올바르게 해석되지 않아 리소스가 일관성 없는 상태로 남을 수 있으므로 팀원은 정리를 실행하지 않아야 합니다.

### 훅으로 품질 게이트 적용

[훅](/en/hooks)을 사용하여 팀원이 작업을 완료하거나 태스크가 완료될 때 규칙을 적용하세요:

* [`TeammateIdle`](/en/hooks#teammateidle): 팀원이 유휴 상태가 되려고 할 때 실행됩니다. 코드 2로 종료하면 피드백을 보내고 팀원이 계속 작업하게 합니다.
* [`TaskCompleted`](/en/hooks#taskcompleted): 태스크가 완료로 표시되려고 할 때 실행됩니다. 코드 2로 종료하면 완료를 방지하고 피드백을 보냅니다.

## Agent teams의 작동 방식

이 섹션에서는 agent teams의 아키텍처와 메커니즘을 다룹니다. 사용 방법을 바로 시작하려면 위의 [Agent team 제어](#control-your-agent-team)를 참조하세요.

### Claude가 agent teams를 시작하는 방법

Agent teams를 시작하는 두 가지 방법이 있습니다:

* **사용자가 팀을 요청**: Claude에게 병렬 작업이 도움이 되는 태스크를 주고 명시적으로 agent team을 요청합니다. Claude가 지시에 따라 팀을 만듭니다.
* **Claude가 팀을 제안**: Claude가 태스크가 병렬 작업의 이점을 받을 수 있다고 판단하면 팀 만들기를 제안할 수 있습니다. 진행하기 전에 사용자가 확인합니다.

두 경우 모두 사용자가 제어권을 유지합니다. Claude는 사용자 승인 없이 팀을 만들지 않습니다.

### 아키텍처

Agent team은 다음으로 구성됩니다:

| 구성 요소     | 역할                                                                                       |
| :------------ | :----------------------------------------------------------------------------------------- |
| **팀 리드** | 팀을 만들고, 팀원을 생성하며, 작업을 조율하는 메인 Claude Code 세션 |
| **팀원** | 할당된 태스크를 각자 처리하는 별도의 Claude Code 인스턴스                            |
| **태스크 목록** | 팀원들이 클레임하고 완료하는 작업 항목의 공유 목록                                |
| **메일박스**   | 에이전트 간 소통을 위한 메시지 시스템                                          |

표시 구성 옵션은 [표시 모드 선택](#choose-a-display-mode)을 참조하세요. 팀원의 메시지는 리드에게 자동으로 전달됩니다.

시스템이 태스크 의존성을 자동으로 관리합니다. 팀원이 다른 태스크가 의존하는 태스크를 완료하면 차단된 태스크들이 수동 개입 없이 차단 해제됩니다.

팀과 태스크는 로컬에 저장됩니다:

* **팀 설정**: `~/.claude/teams/{team-name}/config.json`
* **태스크 목록**: `~/.claude/tasks/{team-name}/`

팀 설정에는 각 팀원의 이름, 에이전트 ID, 에이전트 유형이 포함된 `members` 배열이 있습니다. 팀원들은 이 파일을 읽어 다른 팀원을 발견할 수 있습니다.

### 권한

팀원은 리드의 권한 설정으로 시작합니다. 리드가 `--dangerously-skip-permissions`로 실행되면 모든 팀원도 마찬가지입니다. 생성 후 개별 팀원 모드를 변경할 수 있지만, 생성 시점에 팀원별 모드를 설정할 수는 없습니다.

### 컨텍스트와 소통

각 팀원은 자체 컨텍스트 창을 가집니다. 생성될 때 팀원은 일반 세션과 동일한 프로젝트 컨텍스트(CLAUDE.md, MCP 서버, 스킬)를 로드합니다. 또한 리드의 생성 프롬프트를 받습니다. 리드의 대화 기록은 이어지지 않습니다.

**팀원들이 정보를 공유하는 방법:**

* **자동 메시지 전달**: 팀원이 메시지를 보내면 수신자에게 자동으로 전달됩니다. 리드는 업데이트를 폴링할 필요가 없습니다.
* **유휴 알림**: 팀원이 작업을 마치고 멈추면 자동으로 리드에게 알립니다.
* **공유 태스크 목록**: 모든 에이전트가 태스크 상태를 보고 사용 가능한 작업을 클레임할 수 있습니다.

**팀원 메시지:**

* **message**: 특정 팀원 한 명에게 메시지 전송
* **broadcast**: 모든 팀원에게 동시에 전송. 팀 규모에 따라 비용이 증가하므로 드물게 사용하세요.

### 토큰 사용량

Agent teams는 단일 세션보다 훨씬 많은 토큰을 사용합니다. 각 팀원은 자체 컨텍스트 창을 가지며 토큰 사용량은 활성 팀원 수에 따라 증가합니다. 연구, 검토, 새 기능 작업의 경우 추가 토큰이 대체로 가치가 있습니다. 일상적인 태스크의 경우 단일 세션이 더 비용 효율적입니다. 사용 지침은 [agent team 토큰 비용](/en/costs#agent-team-token-costs)을 참조하세요.

## 활용 사례 예시

이 예시들은 agent teams가 병렬 탐색이 가치를 더하는 태스크를 어떻게 처리하는지 보여줍니다.

### 병렬 코드 리뷰 실행

단일 리뷰어는 한 번에 한 가지 유형의 문제에 집중하는 경향이 있습니다. 리뷰 기준을 독립적인 도메인으로 분리하면 보안, 성능, 테스트 커버리지 모두 동시에 철저한 주의를 받을 수 있습니다. 프롬프트는 각 팀원에게 서로 겹치지 않는 뚜렷한 관점을 할당합니다:

```
PR #142를 검토하는 agent team을 만들어 주세요. 세 명의 리뷰어를 생성해 주세요:
- 한 명은 보안 영향 중점
- 한 명은 성능 영향 확인
- 한 명은 테스트 커버리지 검증
각자 검토하고 발견 사항을 보고하도록 해주세요.
```

각 리뷰어는 동일한 PR에서 작업하지만 다른 필터를 적용합니다. 리드는 모두가 완료된 후 세 곳의 발견 사항을 종합합니다.

### 경쟁 가설로 조사

근본 원인이 불분명할 때, 단일 에이전트는 하나의 그럴듯한 설명을 찾고 더 이상 찾지 않는 경향이 있습니다. 프롬프트는 팀원들을 명시적으로 적대적으로 만들어 이를 방지합니다: 각 팀원의 임무는 자신의 이론을 조사하는 것뿐만 아니라 다른 팀원들의 이론에 도전하는 것입니다.

```
앱이 연결을 유지하지 않고 한 메시지 후 종료된다는 보고가 있습니다.
서로 다른 가설을 조사하는 5명의 agent 팀원을 생성해 주세요.
서로 이야기하여 각자의 이론을 반증하도록 해주세요, 마치 과학적 토론처럼요.
합의가 도출되면 findings 문서를 업데이트해 주세요.
```

토론 구조가 핵심 메커니즘입니다. 순차적 조사는 고착 효과로 어려움을 겪습니다. 하나의 이론이 탐색되면 후속 조사가 그쪽으로 편향됩니다.

서로 적극적으로 반증하려는 여러 독립 조사자들이 있으면 살아남은 이론이 실제 근본 원인일 가능성이 훨씬 높습니다.

## 모범 사례

### 팀원에게 충분한 컨텍스트 제공

팀원들은 CLAUDE.md, MCP 서버, 스킬을 포함한 프로젝트 컨텍스트를 자동으로 로드하지만 리드의 대화 기록을 상속받지는 않습니다. 자세한 내용은 [컨텍스트와 소통](#context-and-communication)을 참조하세요. 생성 프롬프트에 태스크별 세부 사항을 포함하세요:

```
보안 리뷰어 팀원을 다음 프롬프트로 생성해 주세요: "src/auth/에 있는 인증 모듈을
보안 취약점에 대해 검토하세요. 토큰 처리, 세션 관리, 입력 유효성 검사에 집중하세요.
앱은 httpOnly 쿠키에 저장된 JWT 토큰을 사용합니다.
심각도 등급과 함께 문제를 보고하세요."
```

### 태스크 크기 적절히 조정

* **너무 작음**: 조율 오버헤드가 이점을 초과함
* **너무 큼**: 팀원이 체크인 없이 너무 오래 작업하여 낭비된 노력의 위험 증가
* **적절함**: 함수, 테스트 파일, 또는 검토와 같은 명확한 결과물을 생성하는 자체 포함 단위

> **팁**: 리드는 작업을 태스크로 분해하고 팀원에게 자동으로 할당합니다. 태스크가 충분히 만들어지지 않는다면, 작업을 더 작은 단위로 나누도록 요청하세요. 팀원당 5-6개의 태스크가 있으면 모두가 생산적으로 유지되고 리드가 누군가 막히면 작업을 재할당할 수 있습니다.

### 팀원이 완료할 때까지 기다리기

리드가 팀원을 기다리는 대신 직접 태스크를 구현하기 시작하는 경우가 있습니다. 이런 상황이 발생하면:

```
진행하기 전에 팀원들이 태스크를 완료할 때까지 기다려 주세요
```

### 연구 및 검토부터 시작하기

Agent teams를 처음 접하는 경우, 명확한 경계가 있고 코드 작성이 필요하지 않은 태스크(PR 검토, 라이브러리 연구, 버그 조사)부터 시작하세요. 이러한 태스크는 병렬 구현의 조율 문제 없이 병렬 탐색의 가치를 보여줍니다.

### 파일 충돌 방지

두 팀원이 같은 파일을 편집하면 덮어쓰기가 발생합니다. 각 팀원이 서로 다른 파일 집합을 담당하도록 작업을 분리하세요.

### 모니터링 및 방향 조정

팀원들의 진행 상황을 확인하고, 작동하지 않는 접근 방식을 수정하고, 발견 사항이 들어오는 대로 종합하세요. 팀이 너무 오랫동안 감독 없이 실행되면 낭비된 노력의 위험이 증가합니다.

## 문제 해결

### 팀원이 나타나지 않음

Claude에게 팀을 만들도록 요청한 후 팀원이 나타나지 않으면:

* 인프로세스 모드에서 팀원이 이미 실행 중이지만 보이지 않을 수 있습니다. Shift+Down을 눌러 활성 팀원을 순환하세요.
* Claude에게 준 태스크가 팀이 필요할 만큼 복잡했는지 확인하세요. Claude는 태스크에 따라 팀원 생성 여부를 결정합니다.
* 분할 창을 명시적으로 요청한 경우 tmux가 설치되어 PATH에 있는지 확인하세요:
  ```bash
  which tmux
  ```
* iTerm2의 경우 `it2` CLI가 설치되어 있고 iTerm2 설정에서 Python API가 활성화되어 있는지 확인하세요.

### 너무 많은 권한 프롬프트

팀원의 권한 요청이 리드까지 올라와 마찰이 생길 수 있습니다. 팀원을 생성하기 전에 [권한 설정](/en/permissions)에서 일반적인 작업을 미리 승인하여 중단을 줄이세요.

### 오류 시 팀원이 멈춤

팀원이 오류를 만난 후 복구하는 대신 멈출 수 있습니다. 인프로세스 모드에서 Shift+Up/Down을 사용하거나 분할 모드에서 창을 클릭하여 출력을 확인한 다음:

* 직접 추가 지시를 내리거나
* 작업을 계속할 대체 팀원을 생성하세요

### 작업이 완료되기 전에 리드가 종료됨

모든 태스크가 실제로 완료되기 전에 리드가 팀이 완료되었다고 판단할 수 있습니다. 이런 경우 계속하도록 알려주세요. 또한 리드가 위임 대신 작업을 시작하면 팀원들이 완료할 때까지 기다리도록 알려줄 수 있습니다.

### 고아 tmux 세션

팀이 종료된 후 tmux 세션이 지속되면 완전히 정리되지 않았을 수 있습니다. 세션을 나열하고 팀에서 생성한 세션을 종료하세요:

```bash
tmux ls
tmux kill-session -t <session-name>
```

## 제한 사항

Agent teams는 실험적입니다. 알아두어야 할 현재 제한 사항:

* **인프로세스 팀원의 세션 재개 불가**: `/resume`과 `/rewind`는 인프로세스 팀원을 복원하지 않습니다. 세션을 재개한 후 리드가 더 이상 존재하지 않는 팀원에게 메시지를 보내려 할 수 있습니다. 이런 경우 새 팀원을 생성하도록 리드에게 알려주세요.
* **태스크 상태 지연**: 팀원이 태스크를 완료로 표시하지 않아 종속 태스크가 차단되는 경우가 있습니다. 태스크가 막혀 있는 것처럼 보이면 실제로 작업이 완료되었는지 확인하고 태스크 상태를 수동으로 업데이트하거나 리드에게 팀원을 재촉하도록 알려주세요.
* **종료가 느릴 수 있음**: 팀원은 종료 전에 현재 요청 또는 도구 호출을 완료하므로 시간이 걸릴 수 있습니다.
* **세션당 하나의 팀**: 리드는 한 번에 하나의 팀만 관리할 수 있습니다. 새 팀을 시작하기 전에 현재 팀을 정리하세요.
* **중첩 팀 없음**: 팀원은 자체 팀이나 팀원을 생성할 수 없습니다. 리드만 팀을 관리할 수 있습니다.
* **리드는 고정됨**: 팀을 만든 세션이 평생 동안 리드입니다. 팀원을 리드로 승격하거나 리더십을 이전할 수 없습니다.
* **생성 시 권한 설정**: 모든 팀원은 리드의 권한 모드로 시작합니다. 생성 후 개별 팀원 모드를 변경할 수 있지만, 생성 시점에 팀원별 모드를 설정할 수는 없습니다.
* **분할 창에는 tmux 또는 iTerm2 필요**: 기본 인프로세스 모드는 모든 터미널에서 작동합니다. 분할 창 모드는 VS Code의 통합 터미널, Windows Terminal, 또는 Ghostty에서는 지원되지 않습니다.

> **팁**: **`CLAUDE.md`는 정상적으로 작동합니다**: 팀원들은 작업 디렉터리에서 `CLAUDE.md` 파일을 읽습니다. 이를 통해 모든 팀원에게 프로젝트별 지침을 제공할 수 있습니다.

* **경량 위임**: [서브에이전트](/en/sub-agents)는 세션 내에서 연구나 검증을 위한 헬퍼 에이전트를 생성하며, 에이전트 간 조율이 필요하지 않은 태스크에 더 적합합니다

* **접근 방식 비교**: 나란히 비교는 [서브에이전트 vs agent team](/en/features-overview#compare-similar-features)을 참조하세요
