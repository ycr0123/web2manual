# 권한 설정

> 세밀한 권한 규칙, 모드, 관리 정책으로 Claude Code가 접근하고 수행할 수 있는 작업을 제어합니다.

Claude Code는 에이전트가 허용되는 작업과 허용되지 않는 작업을 정확히 지정할 수 있는 세밀한 권한을 지원합니다. 권한 설정은 버전 컨트롤에 체크인하여 조직의 모든 개발자에게 배포하거나 개별 개발자가 커스터마이즈할 수 있습니다.

## 권한 시스템

Claude Code는 권한과 안전성의 균형을 맞추기 위해 계층화된 권한 시스템을 사용합니다:

| 도구 유형 | 예시 | 승인 필요 | "예, 다시 묻지 않음" 동작 |
| :---------------- | :--------------- | :---------------- | :-------------------------------------------- |
| 읽기 전용 | 파일 읽기, Grep | 아니요 | 해당 없음 |
| Bash 명령 | 셸 실행 | 예 | 프로젝트 디렉토리 및 명령별로 영구적으로 |
| 파일 수정 | 파일 편집/쓰기 | 예 | 세션 종료 시까지 |

## 권한 관리

`/permissions`로 Claude Code의 도구 권한을 보고 관리할 수 있습니다. 이 UI는 모든 권한 규칙과 해당 규칙이 출처인 settings.json 파일을 나열합니다.

* **Allow** 규칙은 Claude Code가 지정된 도구를 수동 승인 없이 사용할 수 있도록 합니다.
* **Ask** 규칙은 Claude Code가 지정된 도구를 사용하려 할 때마다 확인을 요청합니다.
* **Deny** 규칙은 Claude Code가 지정된 도구를 사용하지 못하도록 합니다.

규칙은 순서대로 평가됩니다: **deny -> ask -> allow**. 첫 번째로 일치하는 규칙이 적용되므로, deny 규칙이 항상 우선합니다.

## 권한 모드

Claude Code는 도구 승인 방식을 제어하는 여러 권한 모드를 지원합니다. [설정 파일](/en/settings#settings-files)에서 `defaultMode`를 설정하세요:

| 모드 | 설명 |
| :------------------ | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `default` | 표준 동작: 각 도구를 처음 사용할 때 권한을 요청합니다 |
| `acceptEdits` | 세션 동안 파일 편집 권한을 자동으로 수락합니다 |
| `plan` | 플랜 모드: Claude가 파일을 분석할 수 있지만 수정하거나 명령을 실행할 수 없습니다 |
| `delegate` | 에이전트 팀 리더를 위한 조정 전용 모드. 리더를 팀 관리 도구로만 제한하여 모든 구현 작업이 팀원을 통해 이루어집니다. 에이전트 팀이 활성화된 경우에만 사용 가능합니다. 자세한 내용은 [delegate 모드](/en/agent-teams#delegate-mode)를 참조하세요. |
| `dontAsk` | `/permissions` 또는 `permissions.allow` 규칙을 통해 사전 승인되지 않는 한 도구를 자동으로 거부합니다 |
| `bypassPermissions` | 모든 권한 프롬프트를 건너뜁니다 (안전한 환경 필요, 아래 경고 참조) |

> **경고**: `bypassPermissions` 모드는 모든 권한 검사를 비활성화합니다. Claude Code가 피해를 줄 수 없는 컨테이너나 VM과 같은 격리된 환경에서만 사용하세요. 관리자는 [managed 설정](#managed-settings)에서 `disableBypassPermissionsMode`를 `"disable"`로 설정하여 이 모드를 방지할 수 있습니다.

## 권한 규칙 구문

권한 규칙은 `Tool` 또는 `Tool(specifier)` 형식을 따릅니다.

### 도구의 모든 사용과 일치

괄호 없이 도구 이름만 사용하면 도구의 모든 사용과 일치합니다:

| 규칙 | 효과 |
| :--------- | :----------------------------- |
| `Bash` | 모든 Bash 명령과 일치 |
| `WebFetch` | 모든 웹 fetch 요청과 일치 |
| `Read` | 모든 파일 읽기와 일치 |

`Bash(*)`는 `Bash`와 동일하며 모든 Bash 명령과 일치합니다.

### 세밀한 제어를 위한 지정자 사용

괄호 안에 지정자를 추가하여 특정 도구 사용과 일치시킵니다:

| 규칙 | 효과 |
| :----------------------------- | :------------------------------------------------------- |
| `Bash(npm run build)` | 정확한 명령 `npm run build`와 일치 |
| `Read(./.env)` | 현재 디렉토리의 `.env` 파일 읽기와 일치 |
| `WebFetch(domain:example.com)` | example.com에 대한 fetch 요청과 일치 |

### 와일드카드 패턴

Bash 규칙은 `*`를 사용한 glob 패턴을 지원합니다. 와일드카드는 명령의 어떤 위치에도 나타날 수 있습니다. 이 구성은 npm 및 git commit 명령을 허용하면서 git push를 차단합니다:

```json
{
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(git commit *)",
      "Bash(git * main)",
      "Bash(* --version)",
      "Bash(* --help *)"
    ],
    "deny": [
      "Bash(git push *)"
    ]
  }
}
```

`*` 앞의 공백이 중요합니다: `Bash(ls *)`는 `ls -la`와 일치하지만 `lsof`와는 일치하지 않으며, `Bash(ls*)`는 둘 다 일치합니다. 레거시 `:*` 접미사 구문은 ` *`와 동일하지만 더 이상 사용되지 않습니다.

## 도구별 권한 규칙

### Bash

Bash 권한 규칙은 `*`를 사용한 와일드카드 매칭을 지원합니다. 와일드카드는 시작, 중간, 끝을 포함하여 명령의 어떤 위치에도 나타날 수 있습니다:

* `Bash(npm run build)` - 정확한 Bash 명령 `npm run build`와 일치
* `Bash(npm run test *)` - `npm run test`로 시작하는 Bash 명령과 일치
* `Bash(npm *)` - `npm `으로 시작하는 모든 명령과 일치
* `Bash(* install)` - ` install`로 끝나는 모든 명령과 일치
* `Bash(git * main)` - `git checkout main`, `git merge main`과 같은 명령과 일치

`*`가 앞에 공백이 있는 끝에 나타나면 (예: `Bash(ls *)`), 단어 경계를 강제하여 접두사 뒤에 공백이나 문자열 끝이 오도록 요구합니다. 예를 들어 `Bash(ls *)`는 `ls -la`와 일치하지만 `lsof`와는 일치하지 않습니다. 반면 `Bash(ls*)`는 공백 없이 단어 경계 제약이 없으므로 `ls -la`와 `lsof` 모두와 일치합니다.

> **팁**: Claude Code는 셸 연산자(예: `&&`)를 인식하므로 `Bash(safe-cmd *)`와 같은 접두사 매칭 규칙이 있어도 `safe-cmd && other-cmd` 명령을 실행할 수 있는 권한은 부여되지 않습니다.

> **경고**: 명령 인수를 제한하려는 Bash 권한 패턴은 불안정합니다. 예를 들어 `Bash(curl http://github.com/ *)`는 curl을 GitHub URL로 제한하려는 의도이지만, URL 앞에 오는 옵션, 다른 프로토콜, 리다이렉트, 변수, 추가 공백 등의 변형과는 일치하지 않습니다. 더 신뢰할 수 있는 URL 필터링을 위해서는 Bash 네트워크 도구를 제한하거나 URL을 검증하는 PreToolUse hooks를 사용하는 것을 고려하세요.

### Read 및 Edit

`Edit` 규칙은 파일을 편집하는 모든 내장 도구에 적용됩니다. Claude는 Grep 및 Glob과 같이 파일을 읽는 모든 내장 도구에 `Read` 규칙을 최선으로 적용하려 합니다.

Read 및 Edit 규칙은 모두 네 가지 고유한 패턴 유형이 있는 [gitignore](https://git-scm.com/docs/gitignore) 사양을 따릅니다:

| 패턴 | 의미 | 예시 | 일치 |
| ------------------ | -------------------------------------- | -------------------------------- | ---------------------------------- |
| `//path` | 파일 시스템 루트에서의 **절대** 경로 | `Read(//Users/alice/secrets/**)` | `/Users/alice/secrets/**` |
| `~/path` | **홈** 디렉토리에서의 경로 | `Read(~/Documents/*.pdf)` | `/Users/alice/Documents/*.pdf` |
| `/path` | **설정 파일에 상대적인** 경로 | `Edit(/src/**/*.ts)` | `<설정 파일 경로>/src/**/*.ts` |
| `path` 또는 `./path` | **현재 디렉토리에 상대적인** 경로 | `Read(*.env)` | `<cwd>/*.env` |

> **경고**: `/Users/alice/file`과 같은 패턴은 절대 경로가 아닙니다. 설정 파일에 상대적인 경로입니다. 절대 경로를 위해서는 `//Users/alice/file`을 사용하세요.

예시:

* `Edit(/docs/**)`: `<프로젝트>/docs/`에서 편집 (`/docs/`가 아님)
* `Read(~/.zshrc)`: 홈 디렉토리의 `.zshrc` 읽기
* `Edit(//tmp/scratch.txt)`: 절대 경로 `/tmp/scratch.txt` 편집
* `Read(src/**)`: `<현재 디렉토리>/src/`에서 읽기

> **참고**: gitignore 패턴에서 `*`는 단일 디렉토리의 파일과 일치하고 `**`는 디렉토리 전체에 걸쳐 재귀적으로 일치합니다. 모든 파일 접근을 허용하려면 괄호 없이 도구 이름만 사용하세요: `Read`, `Edit`, 또는 `Write`.

### WebFetch

* `WebFetch(domain:example.com)` - example.com에 대한 fetch 요청과 일치

### MCP

* `mcp__puppeteer` - `puppeteer` 서버가 제공하는 모든 도구와 일치 (Claude Code에 구성된 이름)
* `mcp__puppeteer__*` - `puppeteer` 서버의 모든 도구와도 일치하는 와일드카드 구문
* `mcp__puppeteer__puppeteer_navigate` - `puppeteer` 서버가 제공하는 `puppeteer_navigate` 도구와 일치

### Task (서브에이전트)

`Task(AgentName)` 규칙을 사용하여 Claude가 사용할 수 있는 [서브에이전트](/en/sub-agents)를 제어합니다:

* `Task(Explore)` - Explore 서브에이전트와 일치
* `Task(Plan)` - Plan 서브에이전트와 일치
* `Task(my-custom-agent)` - `my-custom-agent`라는 커스텀 서브에이전트와 일치

설정의 `deny` 배열에 이 규칙을 추가하거나 `--disallowedTools` CLI 플래그를 사용하여 특정 에이전트를 비활성화하세요. Explore 에이전트를 비활성화하려면:

```json
{
  "permissions": {
    "deny": ["Task(Explore)"]
  }
}
```

## hooks로 권한 확장

[Claude Code hooks](/en/hooks-guide)는 런타임에 권한 평가를 수행하기 위한 커스텀 셸 명령을 등록하는 방법을 제공합니다. Claude Code가 도구를 호출하면 PreToolUse hooks가 권한 시스템 전에 실행되며, hook 출력으로 권한 시스템 대신 도구 호출을 승인하거나 거부할지 결정할 수 있습니다.

## 작업 디렉토리

기본적으로 Claude는 시작된 디렉토리의 파일에 접근할 수 있습니다. 이 접근을 확장할 수 있습니다:

* **시작 시**: `--add-dir <path>` CLI 인수 사용
* **세션 중**: `/add-dir` 명령 사용
* **영구 구성**: [설정 파일](/en/settings#settings-files)의 `additionalDirectories`에 추가

추가 디렉토리의 파일은 원래 작업 디렉토리와 동일한 권한 규칙을 따릅니다: 프롬프트 없이 읽기 가능하며, 파일 편집 권한은 현재 권한 모드를 따릅니다.

## 권한과 샌드박싱의 상호 작용

권한과 [샌드박싱](/en/sandboxing)은 상호 보완적인 보안 레이어입니다:

* **권한**은 Claude Code가 사용할 수 있는 도구와 접근할 수 있는 파일 또는 도메인을 제어합니다. 모든 도구(Bash, Read, Edit, WebFetch, MCP 등)에 적용됩니다.
* **샌드박싱**은 Bash 도구의 파일 시스템 및 네트워크 접근을 제한하는 OS 수준의 적용을 제공합니다. Bash 명령과 해당 자식 프로세스에만 적용됩니다.

심층 방어를 위해 두 가지를 모두 사용하세요:

* 권한 deny 규칙은 Claude가 제한된 리소스에 접근을 시도하는 것 자체를 차단합니다
* 샌드박스 제한은 프롬프트 주입이 Claude의 의사 결정을 우회하더라도 Bash 명령이 정의된 경계 밖의 리소스에 도달하지 못하도록 합니다
* 샌드박스의 파일 시스템 제한은 별도의 샌드박스 구성이 아닌 Read 및 Edit deny 규칙을 사용합니다
* 네트워크 제한은 WebFetch 권한 규칙과 샌드박스의 `allowedDomains` 목록을 결합합니다

## Managed 설정

Claude Code 구성에 대한 중앙 집중식 제어가 필요한 조직의 경우, 관리자는 시스템 디렉토리에 `managed-settings.json` 파일을 배포할 수 있습니다. 이 정책 파일은 일반 설정 파일과 동일한 형식을 따르며 사용자 또는 프로젝트 설정으로 재정의할 수 없습니다. 기기 관리 인프라가 없는 조직의 경우, [서버 관리 설정](/en/server-managed-settings)이 Anthropic 서버에서 구성을 제공하는 대안을 제공합니다.

**Managed 설정 파일 위치**:

* **macOS**: `/Library/Application Support/ClaudeCode/managed-settings.json`
* **Linux 및 WSL**: `/etc/claude-code/managed-settings.json`
* **Windows**: `C:\Program Files\ClaudeCode\managed-settings.json`

> **참고**: 이들은 관리자 권한이 필요한 시스템 전체 경로(사용자 홈 디렉토리 `~/Library/...`가 아님)로, IT 관리자가 배포하도록 설계되었습니다.

### Managed 전용 설정

일부 설정은 Managed 설정에서만 유효합니다:

| 설정 | 설명 |
| :-------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------- |
| `disableBypassPermissionsMode` | `"disable"`로 설정하면 `bypassPermissions` 모드와 `--dangerously-skip-permissions` 플래그를 방지합니다 |
| `allowManagedPermissionRulesOnly` | `true`로 설정하면 사용자 및 프로젝트 설정이 `allow`, `ask`, `deny` 권한 규칙을 정의하지 못하게 합니다. Managed 설정의 규칙만 적용됩니다 |
| `allowManagedHooksOnly` | `true`로 설정하면 사용자, 프로젝트, 플러그인 hooks 로딩을 방지합니다. Managed hooks 및 SDK hooks만 허용됩니다 |
| `strictKnownMarketplaces` | 사용자가 추가할 수 있는 플러그인 마켓플레이스를 제어합니다. [Managed 마켓플레이스 제한](/en/plugin-marketplaces#managed-marketplace-restrictions)을 참조하세요 |

## 설정 우선순위

권한 규칙은 다른 모든 Claude Code 설정과 동일한 [설정 우선순위](/en/settings#settings-precedence)를 따릅니다: Managed 설정이 가장 높은 우선순위를 가지며, 그 다음은 명령줄 인수, 로컬 프로젝트, 공유 프로젝트, 사용자 설정 순입니다.

사용자 설정에서 권한이 허용되었지만 프로젝트 설정에서 거부된 경우, 프로젝트 설정이 우선하여 권한이 차단됩니다.
