### 🟠 MAJOR-003: SPEC-002 퀴즈 정답 보안 전략 미흡

**심각도**: 🟠 MAJOR
**파일**: SPEC-WEB2MANUAL-002/plan.md
**문제**: 퀴즈 정답을 해시 처리만으로는 리버스 엔지니어링에 취약

**현재 계획**:
```markdown
### 3.3 퀴즈 정답 보안 전략

1. **해시 기반 비교**: 정답 선택지 ID를 SHA-256 해시로 저장
2. **클라이언트 검증 흐름**:
   - 사용자 선택 → 선택지 ID를 해시 → 저장된 해시와 비교
   - 정답 해시만 전송, 원본 정답은 노출하지 않음
3. **해설 표시**: 정답 제출 후에만 해설 텍스트를 표시
```

**문제**:
- SHA-256 해시는 역상 공격에 취약 (해시 값이 노출되면 rainbow table로 복원 가능)
- 4개 선택지 × 3-5 문항 = 최대 20개 해시만 추측하면 됨
- 개발자 도구에서 네트워크 패킷 감시 시 해시 값 노출 가능

**개선 방안**:
```typescript
// 현재 (취약함)
const quizResponse = {
  answered: "option_B",
  answerHash: sha256("option_B"), // 이 값이 노출되면 위험
};

// 개선안
const quizResponse = {
  answered: "option_B",
  // 1. 소금(salt)과 함께 해싱
  answerHash: pbkdf2("option_B" + sessionSalt, iterations),

  // 2. 검증은 서버에서 (SPEC-004 API 연동 필요)
  // 클라이언트에서는 정답을 절대 검증하지 않음
};
```

**해결 방안** (2단계):
1. **Phase 1 (MVP)**: 현재 해시 방식 유지, 하지만 주석에 "MVP 한계"명시
2. **Phase 2 (SPEC-004 완료 후)**: 서버 기반 검증 구현
   - 클라이언트는 답변만 전송
   - 서버가 정답 검증 후 결과 반환
   - 정답 데이터는 서버에서만 보유

**deadline**: SPEC-004 구현 시 함께 처리
